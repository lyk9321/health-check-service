## 01. 프로젝트 배경 및 목표

### 배경

국내 건강검진 수검률은 꾸준히 상승하고 있으나, 결과지를 제대로 활용하는 사람은 소수임. 수치를 받아도 "BMI 25.3이 뭐가 문제인지", "혈압이 높은 건지 낮은 건지" 판단하기 어렵고, 행동으로 이어지는 안내도 부족한 상황임.

특히 20~30대는 혈압·혈당 수치 자체를 모르는 경우가 많아 기존 건강관리 앱의 접근성이 낮았음. 간호학 전공자로서, 의료 도메인 지식과 데이터 분석 역량을 결합해 이 문제를 해결하고자 프로젝트를 기획함.

### 목표

- 건강검진 수치를 모르는 사용자도 BMI만으로 결과를 받을 수 있는 서비스 구현
- 100만 건 실데이터 기반으로 동일 성별·연령대 비교군과의 비교 제공
- 임상 기준(대한비만학회, 대한고혈압학회, 대한당뇨병학회 등)에 근거한 건강관리 유형 6종 분류
- 거주 지역 보건소·건강생활지원센터 정보 연동으로 "다음 행동"까지 연결

## 03. 구현 목표 및 설계 방향

### 핵심 설계: 두 레이어 구조

단일 모델만으로는 개별 지표의 위험 신호와 전체 유형 분류를 동시에 처리할 수 없다고 판단. 두 가지 판단 레이어를 병행하는 구조를 설계함.

```
Layer 1 — 룰 기반 해석 (즉각적)
  사용자 입력 수치 → 임상 기준표 비교
  → BMI·혈압·혈당·콜레스테롤 항목별 판정 문구 출력

Layer 2 — 데이터 기반 분류 (패턴 기반)
  사용자 입력 → Random Forest 분류 모델
  → 건강관리 유형 6종 중 하나 예측
  → 유형별 생활관리 추천 + 비교군 비교
```

두 레이어가 서로 다른 역할을 하므로 결과가 일부 달라도 정상임. 유형 분류는 "전체 수치 패턴의 유사성", 룰 해석은 "개별 지표의 위험 신호"를 담당함.

### 건강관리 유형 6종 정의

| 유형 | 라벨 | 분류 기준 |
|------|------|---------|
| 기본관리형 | `basic` | 위 5가지 모두 해당 없음 |
| 체중관리형 | `weight` | BMI ≥ 25 또는 복부비만 |
| 혈압관리형 | `blood_pressure` | 수축기 ≥ 130 또는 이완기 ≥ 80 |
| 혈당관리형 | `blood_sugar` | 공복혈당 ≥ 100 |
| 대사복합관리형 | `metabolic` | 위험요인 3개 이상 동시 해당 |
| 생활습관관리형 | `lifestyle` | 현재흡연 또는 음주 |

> 우선순위: 대사복합 → 혈압 → 혈당 → 체중 → 생활습관 → 기본  
> "위험형"이 아닌 "관리형" 표현을 사용하여 과잉 진단 오해 방지

## 04. 데이터 전처리 과정

📸 [이미지: 전처리 파이프라인 흐름도 — preprocessing_pipeline.png]

### 처리 순서

**① 컬럼 선택 및 이름 변경**

원본 33개 컬럼에서 분석에 필요한 11개만 선택하고 한국어 컬럼명을 영어로 변환함. 코드 가독성과 유지보수를 위한 기본 정리 단계임.

**② 결측치 처리**

| 변수 유형 | 처리 방법 | 이유 |
|---------|---------|------|
| 성별·연령대·키·몸무게 | 해당 행 제거 | BMI 계산 불가, 서비스 이용 불가 |
| 혈압·혈당·허리둘레 | 성별×연령대 그룹 중앙값 대체 | 비슷한 집단의 대표값으로 합리적 추정 |
| 흡연·음주 | 최빈값 대체 | 범주형 변수이므로 평균 사용 부적절 |

**③ 이상치 제거**

키 100~250cm, 몸무게 20~300kg, 수축기혈압 60~250mmHg 등 의학적으로 불가능한 수치를 가진 행을 제거함.

**④ 파생변수 생성**

- `bmi`: 몸무게(kg) / 키(m)² 계산
- `bmi_category`: 대한비만학회 기준 4단계 분류 (저체중/정상/과체중/비만)
- `hypertension_flag`: 수축기 ≥ 130 또는 이완기 ≥ 80 이면 1
- `hyperglycemia_flag`: 공복혈당 ≥ 100이면 1
- `abdominal_obesity_flag`: 남성 허리 ≥ 90cm, 여성 ≥ 85cm이면 1
- `metabolic_risk_count`: 위 4개 플래그의 합산 (0~4)

## 05. 모델링 및 구현 과정

### 초기 접근: K-Means + Random Forest

처음에는 비지도학습(K-Means)으로 군집을 먼저 만들고, 그 군집 라벨을 정답으로 삼아 Random Forest를 학습하는 2단계 파이프라인을 설계함.

**K-Means 군집화 결과**

📸 [이미지: Elbow Curve + Silhouette Score by K — kmeans_elbow_silhouette.png]

K=6으로 군집화 후 각 군집에 임상 기준으로 유형명을 부여했으나 다음 문제가 발생함:
- Silhouette Score 0.21 (목표 0.30 미달)
- 군집 1: BMI 22.48 (정상 범위)이 체중관리형으로 분류됨
- 군집 1과 군집 2의 실질적 차이가 음주 여부(0.00 vs 1.00)뿐으로 임상적으로 의미 없는 분리

**K-Means 방식의 근본적 한계 확인**

```
K-Means 철학:   "데이터를 보고 자연스러운 패턴을 찾아줘"
이 프로젝트:     "기본관리형, 체중관리형 등 6개 유형으로 나눠줘"
```

사전 정의된 임상 유형과 비지도학습은 처음부터 충돌하는 설계임을 파악함. Random Forest가 K-Means의 경계선을 따라 학습하다 보니 Test Accuracy 0.978이 나왔지만, 이는 "K-Means 경계선을 얼마나 잘 따라했는가"를 측정한 것으로 임상적 의미가 없었음. Confusion Matrix 대부분의 수치가 0.00~0.01로 나온 것도 이 때문임.

### 방향 전환: 룰 기반 라벨링 → Random Forest

임상 기준으로 직접 라벨을 생성하는 방식으로 전환함.

```python
def assign_labels_by_rule(df):
    conditions = [
        df['metabolic_risk_count'] >= 3,
        (df['systolic_bp'] >= 130) | (df['diastolic_bp'] >= 80),
        df['fasting_glucose'] >= 100,
        (df['bmi'] >= 25) | (df['abdominal_obesity_flag'] == 1),
        (df['smoking_status'] == 3) | (df['drinking'] == 1),
    ]

    choices = [
        'metabolic',
        'blood_pressure',
        'blood_sugar',
        'weight',
        'lifestyle',
    ]

    labels = np.select(conditions, choices, default='basic').tolist()

    return labels
```

이 방식으로 바꾸면:
- BMI 정상인 사람이 체중관리형으로 잘못 분류되는 일이 없어짐
- Confusion Matrix 수치가 임상적으로 의미 있는 값으로 변경됨
- PRD 문서에서 K-Means 수식·Silhouette KPI를 모두 제거하고 라벨 분포 검증으로 교체함

## 06. 주요 결과 및 인사이트

### 모델 성능

| 지표 | 값 | 목표 | 달성 여부 |
|------|------|------|--------|
| Overall Accuracy | ~0.78 | ≥ 0.75 | ✅ |
| Macro F1 | ~0.75 | ≥ 0.65 | ✅ |
| 대사복합관리형 Recall | ~0.73 | ≥ 0.70 | ✅ |

### 비교군 통계

성별 2 × 연령대 14 = 28개 그룹별 BMI·혈압·혈당·허리둘레 평균과 표준편차를 사전 집계해 배포용 CSV(~10KB)로 경량화함. 원본 100만 건 CSV(~50MB)는 배포 서버에 올리지 않음.

### 지역 자원 연동

전국 보건소 및 건강생활지원센터 132개소의 주소·전화·운영시간 정보를 거주 시·도 선택에 따라 즉시 출력함. 이를 통해 검진 결과 확인에서 지역사회 자원 활용까지 연결하는 흐름을 완성함.

### 인사이트

- 35-39세 남성의 음주율이 약 78%에 달해, 음주 미입력 시 그룹 평균을 대입하면 `drinking`이 float(0.78)로 들어가는 버그가 발생할 수 있음을 발견함. 이를 binary로 반올림하는 처리를 `predict.py`에 추가함.
- 콜레스테롤 4종 중 LDL이 가장 임상적으로 중요하므로 LDL → HDL → TG → 총콜레스테롤 순서로 우선순위를 적용하고, LDL·HDL·TG 중 하나라도 입력되면 총콜레스테롤은 표시하지 않는 로직을 설계함.

## 07. 어려웠던 점과 해결

- **문제**: K-Means 군집화 + Random Forest 파이프라인을 설계했으나, BMI 정상인 사람이 체중관리형으로 분류되고 Confusion Matrix가 의미없는 0.97을 출력하는 문제가 발생함.  
  **원인 및 해결**: 비지도학습은 데이터의 자연스러운 패턴을 찾는 알고리즘인데, 사전에 6개 임상 유형이 정의된 상황에서 K-Means를 사용하는 것은 모순된 설계임을 파악함. K-Means를 제거하고 임상 기준 우선순위 규칙으로 직접 라벨을 생성하는 룰 기반 라벨링으로 전환함. PRD 문서(4개 파일)도 코드 수정 전에 먼저 업데이트함.  
  **배운 점**: 모델 선택은 정확도뿐 아니라 문제 정의와의 정합성을 먼저 검토해야 함. "비지도 + 지도 학습을 모두 보여주고 싶다"는 포트폴리오 욕심이 임상적 타당성보다 앞선 결과였음. 또한 설계 변경 시 코드보다 PRD를 먼저 수정하는 원칙을 지키는 것이 중요함을 확인함.

- **문제**: 음주 미입력 사용자에게 그룹 평균 음주율(float)을 그대로 `drinking` 변수에 대입하여, 모델이 0 또는 1이 아닌 0.78 같은 값을 입력받는 버그가 발생함.  
  **원인 및 해결**: 흡연은 50% 기준으로 정수 변환 처리를 했지만 음주는 누락된 일관성 문제였음. `1 if drinking_rate >= 0.5 else 0`으로 반올림 처리를 추가함.  
  **배운 점**: 두 변수를 같은 방식으로 처리해야 한다면 코드 내에서 패턴을 통일하고, 미입력 시 대체 로직을 함수로 묶어 한눈에 확인할 수 있게 관리하는 것이 필요함.

- **문제**: Streamlit Cloud 무료 플랜의 메모리 제한(512MB)으로 학습한 Random Forest 모델이 배포 서버에서 로딩되지 않는 문제가 발생함.  
  **원인 및 해결**: `n_estimators=200, max_depth=None`으로 학습하면 모델 파일이 ~500MB가 됨. `n_estimators=100, max_depth=15`로 하이퍼파라미터를 조정해 ~100MB로 축소함. 성능은 Macro F1 기준 유의미한 차이 없음을 확인함.  
  **배운 점**: 배포 환경 제약을 모델 설계 단계에서 미리 고려해야 함. 정확도와 배포 가능성은 트레이드오프 관계임.

- **문제**: 콜레스테롤 해석 문구에서 "LDL 콜레스테롤이 높습니다", "HDL이 높아 심혈관 보호 효과가 있습니다" 같은 단정형·효능 단정 표현이 포함되어 있었음.  
  **원인 및 해결**: PRD의 금지 표현 목록과 대조했을 때 "~입니다"는 진단에 해당할 수 있고, "보호 효과가 있습니다"는 건강기능식품·의료기기 광고법에서 금지하는 효능 단정 표현과 유사함. 모든 메시지를 "~범위에 해당합니다", "~도움이 될 수 있습니다" 형태로 변경함.  
  **배운 점**: 헬스케어 서비스에서는 기술적 정확성뿐 아니라 표현의 법적·의료적 적절성을 함께 검토해야 함. 간호사 도메인 지식이 이 부분에서 직접적으로 활용됨.

## 08. 한계점 및 개선 방향

- 본 서비스의 건강관리 유형 분류는 임상 진단이 아니라 참고용 통계 분류임. 동일한 입력값이라도 실제 의료적 판단과 다를 수 있으며, 모든 결과 화면에 이를 명시하는 주의문구를 필수 표시함.

- 콜레스테롤 4종은 건강검진 데이터에서 결측률이 ~66%이므로 비교군 통계(성별·연령대별 평균 비교)를 제공하지 못함. 사용자가 직접 입력한 경우에 한해 룰 기반 해석만 제공하는 구조로 설계함.

- 건강검진 데이터에 운동 변수가 없어 운동 정보를 모델 학습에 포함하지 못함. 사용자가 직접 입력한 경우에 한해 보건복지부 신체활동 지침과 비교하는 별도 룰 기반 모듈로 처리함.

- 지역보건의료기관 데이터가 2022년 12월 기준으로, 이후 폐쇄·이전·신설된 기관 정보가 반영되지 않음. 공공데이터포털의 최신 버전으로 주기적 업데이트가 필요함.

- BMI 27.4 + 음주 조합처럼 특정 케이스에서 모델이 대사복합관리형으로 오분류하는 경우가 발생함. 룰 기반 라벨링으로 전환했으나, Random Forest가 학습 데이터의 경계 부근 케이스를 간혹 잘못 분류하는 것은 트리 기반 모델의 근본적 한계임. 향후 LightGBM 등 다른 알고리즘과의 성능 비교 실험이 필요함.

- 현재 배포 시 원본 100만 건 CSV는 제외하고 집계 통계 + 학습 모델만 배포하는 구조임. 모델 재학습이 필요한 경우 별도 환경에서 진행해야 하며, 자동화된 재학습 파이프라인은 미구현 상태임.
